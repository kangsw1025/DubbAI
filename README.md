## 서비스 소개 및 주요 기능

**처리 흐름**

```
파일 업로드 → 음성 전사 (ElevenLabs STT) → 번역 (DeepL) → 더빙 합성 (ElevenLabs TTS) → 재생 / 다운로드
```

**주요 기능**

- 오디오(MP3, WAV 등) 및 비디오(MP4, MOV 등) 파일 더빙 지원
- 한국어, 영어, 일본어, 중국어, 프랑스어, 독일어, 스페인어 7개 언어 선택
- 비디오 파일 입력 시 더빙 음성을 원본 영상에 합성하여 MP4로 출력
- 결과물 재생 및 다운로드
- Google 계정 로그인 (화이트리스트 기반 접근 제어)

## 로컬 실행 방법

**1. 저장소 클론**

```bash
git clone https://github.com/kangsw1025/DubbAI.git
cd DubbAI
```

**2. 패키지 설치**

```bash
npm install
```

**3. 환경변수 설정**

`.env.local` 파일을 생성하고 아래 값을 채워주세요.

```env
TURSO_DATABASE_URL=
TURSO_AUTH_TOKEN=
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
NEXTAUTH_SECRET=
NEXTAUTH_URL=http://localhost:3000
ELEVENLABS_API_KEY=
DEEPL_API_KEY=
```

**4. 개발 서버 실행**

```bash
npm run dev
```

`http://localhost:3000` 에서 확인할 수 있습니다.

## AI 에이전트 활용 노하우

이 프로젝트는 Claude Code와 다양한 에이전트를 적극 활용하여 개발했습니다.
직접 사용하면서 효과적이었던 방법들을 공유합니다.

---

### PDCA 워크플로우

**Plan → Do → Check → Act** 사이클을 에이전트와 함께 돌리는 것이 가장 안정적인 개발 방법입니다. 핵심은 **Plan 단계에서 개발자가 계획을 직접 검토하고 승인한 뒤 구현을 시작**하는 것입니다.

#### Plan — 계획만 세우기

계획을 세우기 전, 먼저 아래 준비 단계를 거치면 결과물의 품질이 크게 올라갑니다.

**0. 사전 준비 — 스킬 & 프롬프트 설정**

```
/brainstorm      # 첫 번째 떠오른 방법이 최선이 아닐 수 있으므로, 여러 접근 방식을 비교하고 최선안을 선택
/deep-interview  # AI가 질문을 던지며 모호한 요구사항을 구체화 — 나중에 방향을 뒤집는 비용을 줄여줌
```

CLAUDE.md에 프로젝트 규칙을 미리 작성해두면 에이전트가 바뀌어도 일관된 코드 스타일이 유지됩니다.

```markdown
# CLAUDE.md 예시
- 서비스 레이어 패턴 사용 (lib/services/)
- 외부 API는 반드시 jest.mock() 처리
- async/await 사용, callback 금지
- 컴포넌트는 UI만, 상태는 Custom Hook에
```

**1. 계획 작성 명령**

```
"/pdca plan {피처명} — 코드는 구현하지 말고 계획만 작성해줘"
"/ralplan — 인터뷰 방식으로 PRD를 먼저 설계해줘, 구현은 하지 마"
```

`코드는 구현하지 말고`, `계획만 작성해줘`, `구현은 하지 마` 키워드로 AI가 바로 코드를 작성하는 것을 막습니다.

에이전트가 작성한 계획을 개발자가 직접 검토하고 수정한 뒤, 확정되면 구현을 요청합니다. `/ralplan`은 요구사항을 질문하며 스스로 정리하는 **소크라테스식 인터뷰** 방식으로, 막연한 아이디어도 구체적인 실행 계획으로 발전시켜 줍니다.

실제 계획서 예시: [PLAN.md](./PLAN.md)

#### Do — 계획 기반 구현

```
"위 계획대로 구현해줘"
"/pdca do {피처명}"
```

계획이 확정된 뒤에만 구현을 시작합니다. `oh-my-claudecode:executor` 에이전트를 사용하면 계획서를 받아 집중 구현합니다.

#### Check — 교차 검증

```
"/pdca analyze {피처명}"   # 설계-구현 Gap 분석
"critic 에이전트로 리뷰해줘"
```

구현한 에이전트에게 자신의 코드를 검증시키면 확증 편향 문제가 생깁니다. 반드시 **다른 에이전트**로 검증하세요.

```
# 구현 후 다른 에이전트로 교차 검증
"방금 구현한 코드를 oh-my-claudecode:critic으로 리뷰해줘"
"bkit:gap-detector로 설계 문서와 구현 간의 차이를 분석해줘"
"superpowers:verification-before-completion — 완료라고 하기 전에 먼저 검증해줘"
```

#### Act — 자동 개선

```
"/pdca iterate {피처명}"   # Gap < 90% 시 자동 반복 수정
```

| 단계 | 스킬 / 에이전트 | 역할 |
|------|----------------|------|
| Plan | `/ralplan`, `/pdca plan` | 요구사항 분석, PRD 작성 (구현 없음) |
| Do | `oh-my-claudecode:executor`, `/pdca do` | 계획 기반 구현 |
| Check | `/pdca analyze`, `oh-my-claudecode:critic`, `bkit:gap-detector` | 교차 검증, Gap 분석 |
| Act | `/pdca iterate`, `/ralph` | 자동 반복 개선 |

### 피처 단위 병렬 처리

하나의 피처를 순차적으로 처리하는 것보다, **독립적인 작업을 병렬로 분배**하면 속도가 크게 향상됩니다.

```
"다음 3가지를 병렬로 처리해줘:
1. DeepL 서비스 레이어 구현
2. ElevenLabs 서비스 레이어 구현
3. 위 두 서비스의 유닛 테스트 작성"
```

Claude Code는 한 메시지에 여러 에이전트 호출을 동시에 실행할 수 있습니다. 의존성이 없는 작업은 항상 병렬로 요청하세요.

**병렬 처리 효과적인 케이스**
- 여러 서비스 파일 동시 구현
- 구현과 테스트 코드 동시 작성
- 여러 컴포넌트 분리 작업

**Git Worktree로 피처 격리**

```
/using-git-worktrees  # superpowers:using-git-worktrees
```

여러 피처를 동시에 개발할 때 각 피처를 별도의 worktree에서 작업하면, 브랜치 간 충돌 없이 에이전트들이 완전히 독립된 환경에서 병렬로 작업할 수 있습니다.

### oh-my-claude (OMC) 활용

**oh-my-claude**는 Claude Code를 강화하는 스킬/에이전트 모음입니다. 설치하면 다양한 전문 에이전트와 워크플로우를 즉시 사용할 수 있습니다.

**주요 이점**

- `/ralplan` — 인터뷰 기반 PRD 설계 및 컨센서스 도출
- `/ralph` — 자동 반복 개선 루프 (목표 달성까지 스스로 수정)
- `/ultrawork` — 대량 작업 병렬 처리 엔진
- `oh-my-claudecode:critic` — 구현물에 대한 다각도 심층 리뷰
- `oh-my-claudecode:executor` — 계획서를 받아 집중 구현하는 전용 에이전트

**설치**

```bash
/omc-setup
```

OMC 없이도 Claude Code는 강력하지만, OMC를 추가하면 반복적인 패턴을 스킬로 저장하고 재사용할 수 있어 장기 프로젝트에서 특히 효과적입니다.

### 더 좋은 결과물 만드는 법

**집중 키워드 활용**

특정 부분에 더 많은 연산을 투자하도록 키워드를 사용하면 결과물 품질이 올라갑니다.

```
"보안 취약점을 심층적으로 분석해줘"
"이 로직의 엣지 케이스를 깊이있게 검토해줘"
"성능 병목 지점을 자세히, 빠짐없이 찾아줘"
"이 아키텍처 결정의 트레이드오프를 철저히 따져줘"
```

> `심층적으로`, `깊이있게`, `자세히`, `철저히`, `빠짐없이` 같은 키워드는 해당 영역에 더 많은 처리를 유도합니다.

**우선순위를 명확히 지정**

```
"CRITICAL(버그·보안) → HIGH(성능) → MEDIUM(품질) 순서로 리뷰해줘"
```

**컨텍스트를 미리 제공**

작업 전 관련 파일을 직접 언급하거나 배경을 설명하면 불필요한 탐색 없이 바로 핵심 작업에 집중합니다.

```
"service layer 패턴을 사용 중이고, lib/services/ 아래에 있어. 이 구조를 유지하면서 구현해줘"
```

**복수 대안 요청**

하나의 답만 요청하면 첫 번째 방법을 그냥 씁니다. 여러 대안을 요청하면 더 나은 선택을 할 수 있습니다.

```
"이 문제를 해결하는 3가지 방법과 각각의 장단점을 알려줘"
"가장 단순한 방법, 가장 성능 좋은 방법, 가장 유지보수하기 좋은 방법을 비교해줘"
```

**"왜"를 먼저 설명하기**

목적을 알면 에이전트가 더 적절한 구현을 선택합니다.

```
# 나쁜 예: "에러 처리 추가해줘"
# 좋은 예: "외부 API 호출 실패 시 사용자에게 명확한 메시지를 보여줘야 해서 에러 처리를 추가해줘"
```

**Extended Thinking 토글 — `Option+T`**

복잡한 아키텍처 결정이나 디버깅 시 `Option+T`로 Extended Thinking을 켜세요. 내부 추론에 더 많은 토큰을 사용해 깊이 있는 분석이 나옵니다. 단순 작업에는 끄는 것이 효율적입니다.

### 토큰 줄이는 법

토큰을 아끼면 더 긴 작업을 이어갈 수 있고, 비용도 절감됩니다.

**1. `/compact` — 컨텍스트 압축**

대화가 길어지면 토큰 사용량이 급격히 늘어납니다. 주기적으로 압축하세요.

```
/compact
```

현재까지의 대화를 요약본으로 압축하여 컨텍스트 창을 확보합니다. 긴 작업 중 맥락을 유지하면서 토큰을 아끼는 가장 효과적인 방법입니다.

**2. CLAUDE.md로 반복 설명 제거**

매번 설명하는 내용은 `CLAUDE.md`에 한 번만 작성해두면 모든 대화에서 자동으로 참조됩니다.

```markdown
# CLAUDE.md 예시
- 서비스 레이어 패턴 사용 (lib/services/)
- 외부 API는 반드시 Mock 처리
- 한국어로 응답
```

**3. 작업 범위를 좁게 지정**

넓은 범위 요청은 불필요한 파일 탐색을 유발합니다.

```
# 비효율
"전체 코드를 리뷰해줘"

# 효율
"lib/services/dubbing.service.ts의 에러 처리 부분만 리뷰해줘"
```

**4. 모델 선택 전략**

| 상황 | 추천 모델 |
|------|----------|
| 간단한 코드 수정, 반복 작업 | Haiku (가장 저렴) |
| 일반 개발 작업 | Sonnet (기본값) |
| 아키텍처 설계, 복잡한 버그 | Opus (가장 강력) |

복잡도에 맞는 모델을 선택하는 것만으로도 비용을 크게 줄일 수 있습니다.
